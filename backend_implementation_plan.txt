@# RetailIQ Backend — Issues & Implementation Plan

## 1. Identified Issues & Technical Debt

### A. Testing & Quality Assurance
- **Critical Issue:** 0% Test Coverage. There are no unit tests, integration tests, or repository tests in the `src/test/java` directory.
- **Impact:** High risk of regressions during refactoring or new feature additions.

### B. Security Vulnerabilities
- **Information Disclosure:** `GlobalExceptionHandler.java` catches generic `Exception.class` and returns `ex.getMessage()` in the API response. 
  - *Note in code:* `// Temporarily expose error message for debugging — remove in production`.
- **Hardcoded Secrets/URLs:** `application.yml` contains a hardcoded JWT secret and local URLs (`http://127.0.0.1:8000`), which limits environment portability.

### C. Architecture & Resilience
- **Synchronous Microservice Calls:** `AnalyticsController` uses a synchronous `RestTemplate` call to the Python service. If the Python service is slow, it blocks Tomcat threads. No circuit breaker (like Resilience4j) is implemented.
- **Incomplete Validation:** While some entities use `@Valid`, a comprehensive validation strategy across all REST endpoints is not uniformly enforced for complex DTO relationships.

### D. Transaction Management
- Missing explicit `@Transactional(readOnly = true)` on read-heavy services (e.g., `AnalyticsService`), which can prevent Hibernate from optimizing read queries and allocating connections efficiently.

---

## 2. Implementation Plan

### Phase 1: Security & Exception Handling Hardening
1. **Sanitize Error Responses:** 
   - Update `GlobalExceptionHandler.handleGeneric` to return a generic "An unexpected error occurred" message in production, logging the actual stack trace internally.
2. **Environment Variables:**
   - Move the JWT Secret in `application.yml` to be strictly loaded from environment variables without a fallback, or use a secure vault.

### Phase 2: Testing Infrastructure
1. **Setup Testing Framework:**
   - Integrate JUnit 5, Mockito, and AssertJ (already in spring-boot-starter-test).
2. **Write Core Tests:**
   - **Unit Tests:** `AuthService`, `AnalyticsService`.
   - **Integration Tests:** `ProfileController` and `OrderController` using `@SpringBootTest` and `MockMvc`.
   - **DataJPATests:** Ensure custom queries in `OrderRepository` and `CustomerRepository` are tested against an in-memory DB (H2) or Testcontainers.

### Phase 3: Resilience & Inter-Service Communication
1. **Refactor RestTemplate:**
   - Replace `RestTemplate` with `WebClient` or `RestClient` (Spring Boot 3.2+) for non-blocking calls to the Python analytics service.
2. **Add Circuit Breaker:**
   - Implement Resilience4j to wrap the call to `/analytics/sales`. Return a cached default forecast or handle the timeout gracefully rather than hanging.

### Phase 4: Data Integrity and Optimization
1. **Transaction Tuning:**
   - Audit all Service classes. Add `@Transactional(readOnly = true)` to all data retrieval methods.
2. **DTO Validation Review:**
   - Ensure all incoming POST/PUT request DTOs have `@NotNull`, `@Size`, etc., and Controllers use the `@Valid` annotation.
