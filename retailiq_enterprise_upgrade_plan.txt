================================================================================
  RETAILIQ — ENTERPRISE UPGRADE IMPLEMENTATION PLAN
  Version 2.0 | Transitioning from Reporting Tool to Full ERP/SaaS Platform
================================================================================

EXECUTIVE SUMMARY
─────────────────
The current RetailIQ system is a powerful Data Analytics and Business Intelligence 
(BI) platform. It excels at bulk data ingestion (ETL), machine learning forecasting, 
and KPI reporting.

However, to become a commercially viable SaaS product or a complete Enterprise 
Resource Planning (ERP) system, it requires interactive business operations.

This implementation plan outlines the exact technical steps needed to add:
1. Full Manual CRUD (Create, Read, Update, Delete) Operations
2. Enterprise-Grade Role-Based Access Control (RBAC)
3. Automated System Audit Logs

================================================================================
PHASE 1: ROLE-BASED ACCESS CONTROL (RBAC)
================================================================================

OBJECTIVE: Ensure users only have access to the data and actions necessary for 
their specific job function, protecting the system from unauthorized changes.

Roles to Implement:
1. ADMIN   - Full system access, can manage user accounts.
2. MANAGER - Can perform CRUD operations (edit products/customers).
3. ANALYST - Can trigger bulk data uploads and view reports.
4. VIEWER  - Read-only dashboard access (Executives/Investors).

STEP 1: Backend Security (Spring Boot & MySQL)
  [ ] Update the `users` table to support the new roles as an ENUM or lookup table.
  [ ] Configure Spring Security to enforce method-level security.
      - Add `@PreAuthorize("hasAnyRole('ADMIN', 'MANAGER')")` to all POST/PUT/DELETE 
        endpoints in `ProductController`, `CustomerController`, and `OrderController`.
      - Add `@PreAuthorize("hasRole('ADMIN')")` to the `UserController`.
  [ ] Update the JWT token payload (in `JwtService.java`) to prominently include 
      the user's role upon login.

STEP 2: Frontend Route Protection (React)
  [ ] Define a global state or Context (e.g., `AuthContext`) to store the user's role 
      extracted from the JWT token.
  [ ] Create a `<ProtectedRoute allowedRoles={['ADMIN', 'MANAGER']}>` wrapper component 
      to prevent unauthorized users from even opening certain pages.
  [ ] Conditionally render UI elements based on roles.
      - Example: The "Add Product" button is hidden if `role === 'VIEWER'`.
      - Example: The "Upload Data" sidebar link is hidden if `role === 'MANAGER'`.

================================================================================
PHASE 2: MANUAL CRUD OPERATIONS (Products & Customers)
================================================================================

OBJECTIVE: Allow Operations Managers to manually correct stock quantities, update 
product pricing, and manage individual customer profiles directly from the UI.

STEP 1: Backend API Verification (Java Spring Boot)
  [ ] Audit `ProductController.java` to ensure full endpoint availability:
      - `POST /api/v1/products` (Create)
      - `PUT /api/v1/products/{id}` (Full Update)
      - `PATCH /api/v1/products/{id}/stock` (Partial Update - Quick Stock Adjust)
      - `DELETE /api/v1/products/{id}` (Soft Delete - sets is_active to false)
  [ ] Audit `CustomerController.java` to ensure full endpoint availability:
      - `POST /api/v1/customers` (Create)
      - `PUT /api/v1/customers/{id}` (Full Update)
      - `DELETE /api/v1/customers/{id}` (Soft Delete / GDPR Compliance)
  [ ] Ensure Data Transfer Objects (DTOs) have strict validation (e.g., 
      `@PositiveOrZero` for prices, `@NotBlank` for names).

STEP 2: Frontend Forms & Architecture (React + shadcn/ui)
  [ ] Install `react-hook-form` and `zod` for strict frontend form validation.
  [ ] Create `api.ts` utility methods for the new endpoints using Axios.
  [ ] Create `ProductFormModal.tsx` using the `shadcn/ui` Dialog component.
      - Fields: Name, SKU, Category, Cost Price, Retail Price, Initial Stock.
      - Must handle both "Create" (empty form) and "Update" (pre-filled form) states.
  [ ] Create `CustomerFormModal.tsx` using the `shadcn/ui` Dialog component.
      - Fields: First Name, Last Name, Email, Phone, City, Country.

STEP 3: Implementing CRUD on the Data Tables
  [ ] Update `ProductsPage.tsx`:
      - Add a primary "Add New Product" button at the top right.
      - In the data table, add an "Actions" column with a `DropdownMenu`.
      - Actions: `Edit Product`, `Adjust Stock`, `Deactivate Product`.
  [ ] Update `CustomersPage.tsx`:
      - Add an "Add Customer" button at the top right.
      - In the data table, add an "Actions" column with a `DropdownMenu`.
      - Actions: `Edit Customer Info`, `Delete Customer`.
  [ ] Implement state refresh logic so tables automatically re-fetch data from 
      the API immediately after a successful Create/Update/Delete action.
  [ ] Add Success/Error `Toast` notifications using `shadcn/ui` to give users 
      clear feedback after every operation.

================================================================================
PHASE 3: AUDIT LOGGING & TRACKING
================================================================================

OBJECTIVE: Because manual CRUD operations are now enabled, the system must 
track *who* changed *what* and *when* for security and accountability.

STEP 1: Backend Implementation (Spring Boot)
  [ ] Ensure the `audit_log` table exists in the MySQL database.
  [ ] Create a standard `AuditService` in Spring Boot.
  [ ] In every `PUT` or `DELETE` controller method, intercept the request to 
      log the action.
      - Example Log: "User Admin (ID: 1) changed Unit Price of Product ID 412 
        from 15.00 to 18.00 at 2026-03-12 14:00:00."

STEP 2: Frontend Implementation (React)
  [ ] Create a new `AuditLogPage.tsx` (accessible only by `ADMIN`).
  [ ] Build a read-only table displaying the chronological history of all system 
      changes, sortable by User, Action Type, and Timestamp.
